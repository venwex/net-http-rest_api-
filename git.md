# 2. Основы Git
## 2.1 Создание Git-репозитория
`$ git init`\
Эта команда создаёт в текущем каталоге новый подкаталог с именем .git, содержащий все необходимые файлы репозитория — структуру Git репозитория. На этом этапе ваш проект ещё не находится под версионным контролем

`$ git clone <url>`\
Эта команда позволяет нам скопировать репозиторий с сервера, то есть мы получаем не просто клон а целый репозиторий со всей его историей коммитов, изменений и по сути при потере некоторых данных на сервере наш репозиторий может стать сервером с помощью которого другие разработчики смогут клонить репо. 

## 2.2 Запись изменений в репозиторий
`$ git add`\
Эта команда необходима для того чтобы проиндексировать, то есть добавить все отслеживаемые измененные файлы в индекс, в такой state где позже мы сможем их закомитить. 

`$ git status`\
Эта команда необходима чтобы узнать статус текущей ветки и их файлов, какие были измененые, какие файлы отслеживаемые и нет и т.д.

`$ git diff`\
Если информации из **git status** недостаточно, и вы хотите увидеть конкретные изменения в файлах, используйте эту команду. git diff показывает разницу между рабочей директорией и индексом — то есть, что изменено, но ещё не проиндексировано (не добавлено в git add).\
`$ git diff --staged`\
 Показывает разницу между индексом и последним коммитом — то есть, что вы уже проиндексировали (git add) и что будет включено в следующий коммит.

`$ git commit`\
Все что находится в индексе на момент коммита запоминается гитом, можно сказать что коммит это как точка сохранения в играх к которому можно вернуться. Важно понять что коммиты это по сути снимки экрана твоего проекта на определенный момент разработки, вот и все.\
`$ git commit -a`\
Позволяет без индексации сразу сделать коммит, быстро и удобно, однако стоит быть осторожным потому что флажок **-a** добавляет все измененные файлы в коммит.

`$ git rm`\
Удаляет файл из рабочей директории и из индекса (staging area).То есть файл удаляется физически с диска и будет удалён в следующем коммите.
`$ git rm --cached`\
Удаляет файл только из индекса, но оставляет его на диске.
Это означает, что Git больше не будет отслеживать этот файл.

`$ git mv file_from file_to`\
Переименовывает или перемещает файл с сохранением истории изменений.

## 2.3 Основы Git - Просмотр истории коммитов (все про git log)
`$ git log`\
Эта команда используется для того чтобы просматривать историю коммитов обратном к хронологическому порядке — последние коммиты находятся вверху.

## 2.4 Основы Git - Операции отмены
`$ git commit --amend`\
Эта команда необходима для того чтобы дополнить предыдущий коммит (на самом деле полностью заменяет предыдущий), то есть если мы забыли проиндексировать и добавть какие то файлы в коммит то мы можем их проиндексировать и отправить в коммит с помощью этой команды.

`$ git reset HEAD <file>`\
Эта команда необходима для того чтобы отменить индексацию файла, если мы по ошибке добавили какой то файл с помощью git add то легко можем удалить его из индекса с помощью этой команды.(Не путай со следующей командой, ресет помогает убрать файл из индекса а следующая наоборот отменяет изменения в файле)

`git checkout -- <file>`\
Эта команда позволяет удалить то есть отменить все изменения в файле и откатить до момента последнего коммита этот файл или к началу после клонирования репо.

## 2.5 Основы Git - Работа с удалёнными репозиториями
`git remote`\
Эта команда выведет названия доступных удалённых репозиториев. Если вы клонировали репозиторий, то увидите как минимум origin — имя по умолчанию, которое Git даёт серверу, с которого производилось клонирование.

`git remote add <shortname> <url>`\
Добавляет удаленный репозиторий в конфиги текущего репо

`$ git fetch [remote-name]`\
Эта команда связывается с указанным удалённым проектом и забирает все те данные проекта, которых у вас ещё нет. Важно отметить, что команда\
**git fetch** забирает данные в ваш локальный репозиторий, но не сливает их с какими-либо вашими наработками и не модифицирует то, над чем вы работаете в данный момент.

`$ git pull`\
Эта команда позволяет получить все данные с удаленного репо и слить их с текущей веткой. Главная разница с fetch, в том что, пулл мерджить за нас а в фетч мы сами выбираем что сливать.

`$ git push <remote-name> <branch-name>`\
Эта команда позволяет нам отправить все данные на удаленный сервер которые мы закоммитили. 

`$ git remote show <remote>`\
Показывает больше информации об одном из удаленных репо

`$ git remote show`\
Данная команда показывает какая именно локальная ветка будет отправлена на удалённый сервер по умолчанию при выполнении **git push.** Она также показывает, каких веток с удалённого сервера у вас ещё нет, какие ветки всё ещё есть у вас, но уже удалены на сервере, и для нескольких веток показано, какие удалённые ветки будут в них влиты при выполнении **git pull.**
